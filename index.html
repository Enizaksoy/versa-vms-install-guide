1. Overview

Versa VMS 5.2.2 Versa Director FlexVNF (VOS) AD CS WMI Agent Kubernetes ESXi 7.0

This guide walks the deployment of a standalone Versa Messaging Service (VMS) instance on VMware ESXi, its registration with an existing Versa Director, the configuration of a FlexVNF branch as a passive-authentication consumer, and the installation of a Windows WMI Agent against an Active Directory domain controller. The end state is a live user-IP mapping flowing from AD login events through the WMI Agent to VMS, on to the FlexVNF branch's user-identification table, ready for user-aware policy enforcement.

The sequence is preserved in operator order. Every issue surfaced during the build is documented with the recovery procedure that resolved it — the document doubles as a runbook for the next install and a postmortem of the failure modes that surface in a non-trivial environment.

Final State Branch A-P-HA1 reports live mapping 192.168.30.254 → eniz@mylab.com with status Live in show orgs org-services VERSA user-identification live-users list brief.

What's covered

  • Pre-flight — ESXi sizing, disk expand, network plan (Northbound/Southbound, Elastic VIP), DNS/NTP
  • VMS install — OVA boot, password change (19-char policy), DB cert, server CSR, AD CS web enrollment, import-certs.sh, pre-upgrade patch, vsh configure (Pass 1/2/3)
  • Cert workaround for the message-server pod (internal CA chain rebuild)
  • Director — VMS Connector wizard, branch service profile, tag-value mapping
  • VOS branch (A-P-HA1) — VERSA-Control-VR routing instance, gRPC over overlay, Connected state
  • WMI Agent v11.4.11 install + AD config + sample event flow
  • Verification — eniz@mylab.com → 192.168.30.254 mapping live on the branch
  • Issues hit during the install and how each was fixed (CockroachDB clock skew, message-server CrashLoopBackOff, Director "Cannot reach VMS at IP", Branch "Connecting" indefinitely, etc.)

3.3 Password & Sudoers

VMS blocks vsh status until /opt/versa/etc/install/initial_pass_change exists. The flag is created by a successful vsh modify-system-password run, which changes both admin and versa users in one pass. Policy: minimum 16 characters, no dictionary words, must differ from previous by ≥8 characters.

NOPASSWD sudoers (lab-only, simplifies paramiko-based automation):

echo 'admin ALL=(ALL) NOPASSWD: ALL' | sudo tee /etc/sudoers.d/99-admin-nopasswd
sudo chmod 440 /etc/sudoers.d/99-admin-nopasswd
sudo visudo -c -f /etc/sudoers.d/99-admin-nopasswd
Bug If the password-change script disconnects after admin succeeds, the versa user remains on default and the flag file is never created. Workaround: chpasswd versa user manually, then touch /opt/versa/etc/install/initial_pass_change.

AD CS web enrollment from workstation

Microsoft AD CS web enrollment at http://192.168.20.254/certsrv/ with NTLM authentication, template WebServer. Driven from workstation since VMS itself cannot reach the AD CS LAN directly:

session = requests.Session()
session.auth = HttpNtlmAuth('Administrator', '<ad-admin-pass>')

form = {
    'Mode': 'newreq',
    'CertRequest': open('server-csr.pem').read(),
    'CertAttrib': 'CertificateTemplate:WebServer',
    'TargetStoreFlags': '0', 'SaveCert': 'yes',
    'ThumbPrint': '', 'FriendlyType': 'Saved-Request Certificate',
}
r = session.post('http://192.168.20.254/certsrv/certfnsh.asp', data=form)
req_id = re.search(r'ReqID=(\d+)', r.text).group(1)

signed = session.get(f'http://192.168.20.254/certsrv/certnew.cer?ReqID={req_id}&Enc=b64').text
open('server-cert.pem','w').write(signed)

root = session.get('http://192.168.20.254/certsrv/certnew.cer?ReqID=CACert&Renewal=0&Enc=b64').text
open('root-ca-cert.pem','w').write(root)
Tip The CA template name in the URL is WebServer (no space) although the GUI label is "Web Server". Inspect /certsrv/certrqxt.asp source for <Option Value="O;WebServer;..."> to discover the correct token.

Cert import to VMS

printf '/tmp/root-ca-cert.pem\n\n\n' | sudo /opt/versa/scripts/import-certs.sh -type ca-certs
printf '/tmp/server-cert.pem\n/tmp/server-cert-bundle.pem\n/opt/versa/vms/certs/server-key.pem\n' \
  | sudo /opt/versa/scripts/import-certs.sh -type server-certs
Bug import-certs.sh contains a bash variable name with a hyphen (ca-cert-file-location) which evaluates as $ca + literal "-cert-file-location". root-ca-cert.pem never copies to its target. Manual fix: 
sudo cp /opt/versa/vms/certs/ca-cert.pem /opt/versa/vms/certs/root-ca-cert.pem
sudo touch /opt/versa/vms/certs/imported/root-ca-cert.pem

3.8 Cert Workaround (when imported certs cause pod crash)

Used when AD-CS-signed certs are required but the standard vsh configure path with regenerate=N has left message-server in CrashLoopBackOff:

# Generate fresh internal CA chain (root + intermediate)
sudo /opt/versa/vms/certs/vms_cert_gen.sh \
  --domain vms-1.mylab.com --country TR --state Istanbul --locality Istanbul \
  --organization VersaLab --organizationalunit IT \
  --email admin@example.com \
  --keypass '<keypass>' --validity 3650 \
  --san vms-1.mylab.com,DNS.1:vms-1-elastic.mylab.com,DNS.2:vms-vos.mylab.com,DNS.3:vms-1,IP.1:192.168.30.15,IP.2:192.168.30.16,IP.3:192.168.20.15

# Decrypt ca-key (openssl ca needs unencrypted private key)
sudo openssl rsa -in /opt/versa/vms/certs/ca-key.pem \
  -out /tmp/ca-key.pem -passin pass:'<keypass>'
sudo cp /tmp/ca-key.pem /opt/versa/vms/certs/ca-key.pem
sudo chmod 400 /opt/versa/vms/certs/ca-key.pem

# Sign existing CSR with the new internal CA
sudo /opt/versa/scripts/certificates/sign-server-cert-with-local-CA.sh \
  /opt/versa/vms/certs/server-csr.pem

# Rename "secondary-*" outputs to active filenames
sudo cp /opt/versa/vms/certs/secondary-server-cert.pem        /opt/versa/vms/certs/server-cert.pem
sudo cp /opt/versa/vms/certs/secondary-server-cert-bundle.pem /opt/versa/vms/certs/server-cert-bundle.pem
sudo chmod 644 /opt/versa/vms/certs/server-cert.pem /opt/versa/vms/certs/server-cert-bundle.pem /opt/versa/vms/certs/server-key.pem

# Restart pod
sudo kubectl delete pod -n message-server -l app=message-server

Verify the bundle has 3 BEGIN markers (server + intermediate + root): grep -c BEGIN /opt/versa/vms/certs/server-cert-bundle.pem should output 3.

4. Director VMS Connector

4.1 Clock Sanity

VMS, Director, and the AD/CA infrastructure must agree on real time within seconds. CockroachDB inside VMS uses an HLC and refuses operations when wall time drifts more than a few seconds across nodes or jumps backward across its own stored timestamps. Symptom in cockroach.log if violated:

"unable to get liveness record from KV: aborted in DistSender: context deadline exceeded"
"ERROR: remote wall time is too far ahead (32h…) to be trustworthy"

Director's PUT /vmsadmin/commonconfigs returns HTTP 500 in this state, and the connector wizard's Submit fails with "UPDATE FAILED".

4.2 CockroachDB Recovery (only if clock skew bricked the DB)

Bug The CockroachDB store is at the literal directory /$DB_STORE$NODE (with the dollar signs as part of the directory name) because systemd EnvironmentFile does not perform nested variable expansion. The systemd unit's --store=${NODE_STORE} argument is read as the literal string $DB_STORE$NODE. Wiping /var/lib/db/data/* alone leaves the real data directory intact. 
systemctl stop vms-admin vms-db
rm -rf '/$DB_STORE$NODE' /var/lib/db/data/*
systemctl start vms-db
sleep 25
/usr/local/bin/cockroach init \
  --certs-dir=/opt/versa/vms/certs/db_certs/ \
  --host=localhost:26257 --cluster-name=vmsadminconfig
/usr/local/bin/cockroach sql \
  --certs-dir=/opt/versa/vms/certs/db_certs/ \
  --host=localhost:26257 < /opt/versa/vms/db/db_script.sql
systemctl start vms-admin

7. End-to-End Verification

Live Mapping 
admin@A-P-HA1-cli> show orgs org-services VERSA user-identification live-users list brief

IP ADDRESS      NAME            STATUS  HITS  EXPIRY  MODE
192.168.30.254  eniz@mylab.com  Live    0     53036   inactivity

Pipeline trace

  1. AD login event — eniz logs into a domain-joined machine. AD generates a 4624 (Logon) event on the DC.
  2. WMI Agent reads the AD security event log, looks up source IP, packages the event as User=eniz@mylab.com, IP=192.168.30.254, Event=LOGIN, Tenant=Versa.
  3. WMI → VMS over HTTPS POST to https://vms-1-elastic.mylab.com:443/v1/tenants/VERSA/... arriving at MetalLB-managed Elastic VIP 192.168.30.16:443 and routed through ingress-nginx to the third-party-authentication pod.
  4. third-party-auth pod applies the tenant tag map (mylab.com → VERSA) and publishes to message-server topic passive-auth-uip for tenant VERSA.
  5. message-server pod stores the event and pushes it on the active gRPC subscription stream over port 1376 to the branch.
  6. FlexVNF updates its user-identification live-users table; subsequent traffic from 192.168.30.254 matches user eniz@mylab.com for NGFW, SSE, and SD-WAN policies.

8. Issues Encountered & Resolutions

Issue 1 — Disk pre-check fails

Symptom: vsh configure aborts with "Server requirements not met. Disk: 146 GiB".
Fix: Expand virtual disk on ESXi to ≥1 TB; growpart /dev/sda 1 + LVM extend + resize2fs.

Issue 2 — Default-password warning blocks vsh status

Symptom: Banner says "System contains default/expired passwords" even after admin password change.
Fix: Run vsh modify-system-password to completion (it changes both admin AND versa users). The flag /opt/versa/etc/install/initial_pass_change is created on success. If your script disconnects partway, finish versa user with chpasswd and create the flag manually.

Issue 3 — import-certs.sh bash variable bug

Symptom: After vsh import-ca-certs, root-ca-cert.pem not present in /opt/versa/vms/certs/.
Cause: Hyphen in shell variable name (ca-cert-file-location) parses as $ca + literal "-cert-file-location".
Fix: cp ca-cert.pem root-ca-cert.pem + touch imported/root-ca-cert.pem.

Issue 4 — message-server CrashLoopBackOff after vsh configure

Symptom: Pod log: tls: failed to find any PEM data in certificate input. Bundle has 2 BEGIN markers (server + root only).
Cause: User answered N to "Generate cert with FQDN" during Pass 2. VMS kept imported certs but skipped the internal cert chain rebuild.
Fix: Apply the cert workaround in §3.8 — generate fresh internal CA via vms_cert_gen.sh, sign existing CSR with sign-server-cert-with-local-CA.sh, rename outputs, restart pod. Bundle should have 3 BEGIN markers.

Issue 5 — CockroachDB stuck after time jump

Symptom: vms-admin in restart loop with "Failed to obtain JDBC Connection"; cockroach.log floods with "unable to get liveness record from KV".
Cause: Wall time jumped backward across HLC timestamps stored before the jump.
Fix: Stop services; wipe BOTH /var/lib/db/data/* and /$DB_STORE$NODE (literal); start vms-db; cockroach init; apply db_script.sql; start vms-admin.

Issue 6 — Director "Cannot reach VMS at IP" on connector wizard

Symptom: Connector form returns to Step 2 with the error banner. tcpdump on VMS shows zero ingress on port 8091.
Causes (any of): message-server pod in Unknown state with no service endpoints; clock skew preventing CockroachDB writes; vms-admin Spring Boot in restart loop.
Fix: kubectl delete pod -n message-server --all --grace-period=0 --force; if clock issue, recover Cockroach per §4.2.

Issue 7 — Branch shows Connecting indefinitely

Symptom: show orgs org-services VERSA vms status stays at Connecting; branch log shows dial tcp4 192.168.20.15:1376: i/o timeout; tcpdump on VMS captures ICMP from branch but no TCP SYN on 1376.
Causes (any of): message-server pod in Unknown state; branch CA chain missing intermediate; older FlexVNF build incompatible with VMS 5.2.2; Controller's Allow-From-CPE-Ports policy missing 1376.
Fix: Restart message-server pod; rebuild CA chain with both intermediate and root; upgrade FlexVNF; verify Controller policy.

Issue 8 — Branch Connected but Messages received: 0

Symptom: gRPC connection up but no events flowing.
Cause: Director Connector Step 3 Tag Value left blank — VMS has no way to route mylab.com-domain WMI events to the VERSA tenant topic.
Fix: Edit Connector → Step 3 → set Tag Value to mylab.com → Submit.
